# v0.5 Tool Registry — Convention-based Discovery + Stability Tiers

## Overview

resolume-mcp-server v0.4.2 ships 36 tools registered through a manually maintained array in `src/tools/index.ts`. Every new tool requires three coordinated edits: the tool source file, the import block in `index.ts`, and the `eraseTool(...)` entry. A separate parity script (`scripts/check-skill-sync.mjs`) keeps `SKILL.md` in lockstep, and `prepublishOnly` enforces it. This works at 36 tools but is approaching the cliff where contributors forget one of the three sites.

v0.5 introduces two related improvements:

1. **Convention-based discovery** — file naming is the source of truth. Adding `src/tools/<domain>/<verb-noun>.ts` with the right export pattern makes the tool live; no more manual array.
2. **Stability tiers + deprecation** — every `ToolDefinition` carries an explicit lifecycle marker (`stable` | `beta` | `alpha`), with optional structured `deprecated` metadata.

The design preserves three hard constraints:

- **No breaking change to tool authoring.** Existing 36 tools compile unchanged because `stability` defaults to `"stable"` and `deprecated` is optional.
- **Works as published npm package.** Discovery is build-time, not runtime fs glob.
- **`check-skill-sync.mjs` keeps working** with a small consumption-side change.

## Discovery mechanism

### Build-time vs runtime analysis

| Dimension | Build-time codegen | Runtime fs discovery |
|---|---|---|
| **npm publish** | Generated file lands in `build/` like every other compiled artifact. Ships exactly what was tested. | Must `fs.readdir(import.meta.url + "../tools")` against published `build/`. Bundlers (esbuild, ncc) silently drop dynamic `import()` graphs that aren't visible statically. |
| **Cold start** | Zero IO; identical to today (one synchronous import graph). | One `readdir` + N async `import()` per process start (~5–15 ms). |
| **Type safety** | Generated file is real TS that `tsc` checks. Errors surface at build time. | Tools are `await import()`ed as `unknown`, then narrowed with Zod. Errors surface only at server startup. |
| **Dev DX** | `npm run dev` (`tsc --watch`) needs to re-run codegen on file add. | Add a file, restart server, done. |
| **Tree-shakability** | Static imports — bundlers see everything. | Dynamic imports — bundlers either inline everything or break. |
| **Skill-sync impact** | Script reads `tool-manifest.json` emitted alongside. Same regex parsing. | Script must invoke Node to enumerate fs OR duplicate discovery logic — drift risk. |
| **Handling broken tool files** | Codegen fails fast with clear error. Build is red. | Server starts, throws on import, MCP client gets cryptic startup error. |
| **Determinism** | Sorted glob → stable file order → stable `tools/list` output → stable LLM caching. | `fs.readdir` order is FS-dependent. |

### Decision: build-time generation

Build-time wins decisively. Runtime savings are illusory (5–15 ms once at startup); npm-publish + bundler safety story is materially better.

### File / script outline

#### `scripts/gen-tool-index.mjs` (new)

Globs `src/tools/**/*.ts` (excluding `types.ts`, `*.test.ts`, `index*.ts`) and emits two artifacts:

1. **`src/tools/index.generated.ts`** — import graph + `allTools` array. Git-committed (so PR diffs visibly show new tools).
2. **`src/tools/tool-manifest.json`** — `{ tools: [{ name, file, stability, deprecated, destructive }] }`. Consumed by `check-skill-sync.mjs` and tests.

Validation rules (build fails on any violation):
- File must export at least one symbol matching `/^[a-z][A-Za-z0-9]*Tool$/`.
- Each exported tool must have `name === "resolume_<snake_case>"`.
- Each exported tool's file path must encode the same verb-noun: `src/tools/clip/trigger-clip.ts` → `triggerClipTool`, name `"resolume_trigger_clip"`.
- `stability` must be one of `"stable" | "beta" | "alpha"`.
- `deprecated.since` must match a semver string.

Implementation: use Node 22+ `fs.glob`. Regex-extract `export const xxxTool: ToolDefinition<...>` and the `name: "resolume_..."` literal. Walk results in sorted order for deterministic output.

#### `src/tools/index.generated.ts` (new, committed)

```text
// AUTO-GENERATED by scripts/gen-tool-index.mjs — DO NOT EDIT
// Run `npm run gen:tools` to refresh.

import { eraseTool, type AnyTool } from "./registry.js";
import { triggerClipTool } from "./clip/trigger-clip.js";
import { clearClipTool, wipeCompositionTool } from "./clip/clear-clip.js";
// ...

export const allTools: ReadonlyArray<AnyTool> = [
  eraseTool(triggerClipTool),
  eraseTool(clearClipTool),
  eraseTool(wipeCompositionTool),
  // ...
];
```

#### `src/tools/index.ts` (becomes thin re-export)

```text
export { allTools } from "./index.generated.js";
export type { AnyTool } from "./registry.js";
```

#### `src/tools/registry.ts` (new — extracted from current `index.ts`)

Holds `AnyTool` interface, `eraseTool()`, plus new `decorateDescription()`/`filterByStability()` helpers.

#### `package.json` script changes

```text
"scripts": {
  "gen:tools":        "node scripts/gen-tool-index.mjs",
  "build":            "npm run gen:tools && tsc && node -e \"require('fs').chmodSync('build/index.js', 0o755)\"",
  "dev":              "npm run gen:tools && tsc --watch",
  "check:tools":      "node scripts/gen-tool-index.mjs --check",
  "check:skill-sync": "node scripts/check-skill-sync.mjs",
  "prepublishOnly":   "npm run check:tools && npm run build && npm test && npm run check:skill-sync"
}
```

`--check` mode regenerates in memory and diffs against the committed file.

#### Pre-commit hook
Add `npm run gen:tools` to existing pre-commit hook so contributors never forget.

## Naming convention enforcement

Three layers, all enforced by `gen-tool-index.mjs`:

### Layer 1: file path → kebab-case verb-noun
`src/tools/<domain>/<verb-noun>.ts`. Single deterministic transformation: file path → expected camelCase → expected `*Tool$` symbol → expected `name: "resolume_<snake_case>"`.

### Layer 2: export symbol pattern
Each tool file MUST export at least one symbol matching `/^[a-z][A-Za-z0-9]*Tool$/`. Files exporting multiple tools (e.g. `clear-clip.ts` exports both `clearClipTool` and `wipeCompositionTool`) are allowed.

### Layer 3: name literal must match
Inside the file, `name: "resolume_<snake_case>"` must encode the same verb-noun.

### Bypass for outliers
`// gen-tool-index: name-override` directive immediately above the export. Audited at code review. Not expected in v0.5.

### Build-fail examples
```
gen-tool-index: src/tools/clip/triggerClip.ts uses camelCase filename, expected kebab-case (trigger-clip.ts).
gen-tool-index: src/tools/clip/trigger-clip.ts exports triggerClipTool with name "resolume_trigger_clip_v2", expected "resolume_trigger_clip".
```

## Stability tiers

### Type extension (no breaking change)

```text
export type Stability = "stable" | "beta" | "alpha";

export interface DeprecationInfo {
  since: string;
  replaceWith?: string;
  removeIn?: string;
  reason?: string;
}

export interface ToolDefinition<TShape extends z.ZodRawShape = z.ZodRawShape> {
  name: string;
  title: string;
  description: string;
  inputSchema: TShape;
  destructive?: boolean;
  /** Defaults to "stable". */
  stability?: Stability;
  /** Presence implies deprecated. */
  deprecated?: DeprecationInfo;
  handler: ...;
}
```

### Description decoration

`registry.ts` exposes pure `decorateDescription(tool)`:
1. Tier prefix: `[BETA] ` or `[ALPHA] ` (none for stable).
2. Original description.
3. Deprecation suffix: ` (deprecated since X[, use Y][, removed in Z])`.

Example for deprecated alpha tool:
> `[ALPHA] Listens on Resolume's OSC OUT port... (deprecated since 0.5.0, use resolume_osc_subscribe_v2, removed in 0.6.0)`

### Visibility filter

Env var `RESOLUME_TOOLS_STABILITY` controls what's exposed:

| Value | Exposes |
|---|---|
| `stable` | only `stable` |
| `beta` (default) | `stable` + `beta` |
| `alpha` | all three |

In `registerTools.ts`, before iterating `allTools`: `const minTier = parseStability(process.env.RESOLUME_TOOLS_STABILITY ?? "beta")` and filter. Always log to stderr at startup: `resolume-mcp-server: tier filter = beta (1 alpha tool hidden)`.

### Deprecation runtime warning

```text
const warned = new Set<string>();
if (tool.deprecated && !warned.has(tool.name)) {
  warned.add(tool.name);
  process.stderr.write(`[resolume-mcp-server] WARNING: tool '${tool.name}' is deprecated since ${tool.deprecated.since}...\n`);
}
```

Module-scoped `warned` set ensures one warning per tool per process.

### Concrete example: `tap_tempo` → beta

One line added:
```text
export const tapTempoTool = {
  name: "resolume_tap_tempo",
  title: "Tap tempo",
  stability: "beta",                           // ← new
  description: "...",
  ...
};
```

LLM sees `[BETA] ...`. Default deploys still expose it. Risk-averse operator can set `RESOLUME_TOOLS_STABILITY=stable` and it disappears.

## Deprecation lifecycle

Five stages, with worked example `osc_status` → `osc_diagnostics` over v0.5.0/v0.6.0/v0.7.0:

### Stage 1: stable (steady state)
No metadata. v0.4.x state.

### Stage 2: announce — add `deprecated` metadata, keep functional

```text
deprecated: {
  since: "0.5.0",
  replaceWith: "resolume_osc_diagnostics",
  removeIn: "0.7.0",
  reason: "osc_diagnostics returns probe data plus latency histogram",
},
```

CHANGELOG entry under `### Deprecated` block.

### Stage 3: parallel availability
v0.5.x and v0.6.x ship both. SKILL.md updates with separate "Legacy / deprecated" section.

### Stage 4: removal in `removeIn` release
v0.7.0: delete file. `gen-tool-index.mjs` regenerates without it. CHANGELOG `### Removed (BREAKING)`. **Recommend bumping to 1.0.0 before any removal.**

### Stage 5: enforcement gate

```text
test("no tool stays past its removeIn version", () => {
  const pkgVersion = packageJson.version;
  for (const tool of allTools) {
    if (tool.deprecated?.removeIn && semverGte(pkgVersion, tool.deprecated.removeIn)) {
      throw new Error(`${tool.name} should have been removed in ${tool.deprecated.removeIn}`);
    }
  }
});
```

### Worked example timeline

| Version | `osc_status` | `osc_diagnostics` |
|---|---|---|
| 0.4.2 | stable | does not exist |
| 0.5.0 | deprecated{since:0.5.0, removeIn:0.7.0} | stable (new) |
| 0.6.0 | unchanged — warns | stable |
| 0.7.0 | **deleted** | stable |

## check-skill-sync update plan

Switch input from parsing `index.ts` to reading `tool-manifest.json`.

```text
function collectRegisteredToolNames() {
  const manifestPath = join(repoRoot, "src", "tools", "tool-manifest.json");
  if (!existsSync(manifestPath)) {
    throw new Error("check-skill-sync: tool-manifest.json missing — run `npm run gen:tools` first");
  }
  const manifest = JSON.parse(readFileSync(manifestPath, "utf8"));
  return new Set(manifest.tools.map((t) => t.name));
}
```

Everything else (mentioned-name harvesting, count callouts, exit codes) stays identical.

### Bonus: stability-aware skill check (v0.5.1)
Once manifest exposes `stability`, script can enforce: "any beta or alpha tool MUST appear under SKILL.md's `### Beta tools` / `### Alpha tools` section."

### prepublishOnly chain
1. `check:tools` (gen --check — codegen up to date)
2. `build` (gen + tsc)
3. `test` (vitest, 80% coverage gate)
4. `check:skill-sync` (manifest vs SKILL.md)

## Migration plan

### Phase 0 — preparation (single PR, no behavior change)
1. Add `scripts/gen-tool-index.mjs`. Commit `src/tools/index.generated.ts` and `tool-manifest.json`.
2. Run side-by-side: `index.generated.ts` AND existing `index.ts` both export `allTools`. Tests assert identical.
3. Ship as no-op PR. Easy revert.

### Phase 1 — flip the import (PR #2)
1. Change `src/server/registerTools.ts` to import from `./tools/index.generated.js`.
2. Replace `index.ts` body with re-export.
3. Update `check-skill-sync.mjs` to read manifest.
4. Ship as v0.4.3 (patch).

### Phase 2 — stability tiers (PR #3, target v0.5.0)
1. Extend `ToolDefinition` with optional `stability` and `deprecated`.
2. Add `decorateDescription` and `filterByStability`.
3. Add env var doc.
4. Mark `tap_tempo` as `stability: "beta"`.
5. Ship as v0.5.0 (minor — additive surface).

### Phase 3 — first deprecation (PR #4 onwards)
When first tool replaced, follow Stage 2 lifecycle.

### Backward compat
- Existing tool files: zero edits required.
- Existing LLM prompts: tool names unchanged. Descriptions only gain content.
- `prepublishOnly`: still runs `check:skill-sync`, gains one new step.
- `build/` adds two files (`tools/index.generated.js`, `tools/tool-manifest.json`).

### Rollback plan
If Phase 1 breaks: revert PR #2 only. Phase 0's generated file remains harmless.

## Risks

### R1 — Codegen / source drift
Mitigation: file banner + `--check` gate in CI + pre-commit hook running `gen:tools`.

### R2 — Glob picks up unintended files
Mitigation: glob excludes `*.test.ts` and `_*`. Export-shape regex is the actual gate.

### R3 — Bundlers strip generated file
**No mitigation needed** — static imports, bundler-safe.

### R4 — Operator surprise on `RESOLUME_TOOLS_STABILITY=stable`
Mitigation: stderr line at startup ("3 beta tools hidden by RESOLUME_TOOLS_STABILITY=stable") + README documentation.

### R5 — Description length growth from prefixes
Mitigation: not needed at current scale. Reconsider if catalog hits 200+ tools.

### R6 — `manifest.json` merge conflicts
Mitigation: sorted JSON, line-local conflicts. If pain emerges, switch to manifest-per-tool.

### R7 — Regex breaks on a complex tool file
Mitigation: explicit failure modes. Migrate to TS AST parser (`ts-morph`) when needed (1-day fix).

### R8 — `removeIn` enforcement test too aggressive
Mitigation: failing test forces conversation. Fix is one-line bump.

### R9 — Skill drift on stability tiers
Mitigation: addressed in v0.5.1 by stability-aware skill check.

## Executive summary

- **Pick build-time codegen** (`scripts/gen-tool-index.mjs` → `src/tools/index.generated.ts` + `tool-manifest.json`) over runtime fs discovery — bundler-safe, works in published `build/`, fails fast at build time, and reduces `check-skill-sync.mjs` complexity by switching its input to the manifest.
- **Naming convention is enforced at three layers** — file path → kebab-case verb-noun, exported symbol matching `*Tool$`, `name:` literal matching the path-derived `resolume_<snake_case>` — with a `// gen-tool-index: name-override` escape hatch for legacy aliases. Build fails with file-precise errors on any mismatch.
- **Stability tiers are additive and backward-compatible**: `stability?: "stable" | "beta" | "alpha"` (default `stable`) decorates the description with a `[BETA]`/`[ALPHA]` prefix; `RESOLUME_TOOLS_STABILITY` env var (default `beta`) lets operators hide alpha tools without forking. `tap_tempo` becomes a one-line change to mark it beta.
- **Deprecation is a structured 5-stage lifecycle** — announce (`deprecated{since,replaceWith,removeIn,reason}`) → parallel availability → first-call stderr warning (once per process) → removal in `removeIn` → CI test asserting no tool overstays its `removeIn` version. Worked example: `osc_status` → `osc_diagnostics` over 0.5.0/0.6.0/0.7.0.
- **Migration is three small PRs**: Phase 0 lands codegen alongside the manual array (no behavior change, easy revert), Phase 1 flips the import and trims `index.ts` to a re-export (ships as v0.4.3 patch), Phase 2 adds stability/deprecation types and the env filter (ships as v0.5.0 minor). Existing 36 tools and consumer prompts are unaffected.
